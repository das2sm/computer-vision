Bilinear Interpolation:
Accurate Feature Alignment: It ensures that the query accurately captures

By using the Homogeneous Counterpart c*li = cli <append> 1 (to match

1. The Core Problem: The "Ray" of Ambiguity

When a 2D camera sees a red pixel, it knows the direction of that red object, but it doesn't know the distance. In 3D math, that pixel represents an infinite ray stretching out from the lens.

    The Challenge: To move to the "Ego Frame" (3D), we need depth. But depth is "inherently ambiguous" in a single photo.

2. The Solution: The "Frustum" Point Cloud

Instead of guessing one depth, LSS creates a Frustum (a pyramid-shaped 3D volume) for every camera.

    The Grid: For a single image of size H×W, they define a set of discrete depth "bins" D (e.g., every 1 meter from 0m to 100m).

    The Points: They create D×H×W points.

        If your image is 200×100 and you have 100 depth bins, you now have 2 million points in a 3D cloud for just one camera.

    No Learning here: This part is just pure geometry. It’s like placing 100 transparent sheets of glass in front of the camera.

3. The "Lift" Equation: cd=αdc (Eq. 1)

This is how the model "fills" that 3D cloud with intelligence. For every single pixel p, the neural network (the ResNet/Backbone) predicts two things:

    Context (c): A vector of features (C-channels). This represents "What is this?" (e.g., "This looks like a blue car door").

    Depth Distribution (α): A list of probabilities for every depth bin. This represents "Where is this?" (e.g., "I am 10% sure it's at 5m, 80% sure it's at 10m, and 10% sure it's at 15m").

The "Lift" Action: The feature at a specific depth d (cd) is simply the What (c) multiplied by the Where (αd).

    If the probability (αd) is near zero for a certain depth, that part of the 3D cloud stays "empty."

    If the probability is high, that part of the 3D cloud gets "filled" with the car door features.

Splatting - Using pillars - squashes 3D point clouds into 2D BEV - then use 2D CNNs for reasoning. No LiDAR needed at any point to verify depth, verification done in 2D - cumsum trick

Shooting - Picks lowest cost movement based on a library of template trajectories (made using K-means) - makes AI interpretable

Sum pooling: Evidence: If Camera 1 sees "0.4 Car-ness" and Camera 2 sees "0.5 Car-ness" at the same spot, the Sum (0.9) tells the planner: "We are now very sure there is a car here because multiple sensors agree." An "Average" would just stay at ~0.45, losing that boost in confidence.


nuScenes
Every 2Hz, 0.5s - a snapshot is taken of all sensors - contains attributes - visibility, activity, pose
Dataset also contains intermediate frames - used for tracking and prediction
Annotations only done if hit by at least lidar or radar - so annotations are backed by real sensor data not just images

Detection metric:
AP:
Instead of IOU - uses 2D center distance on ground plane - smaller objects give 0 IOU if off by even a little
AP is calculated as area under precision-recall curve, clipping the first 10% on both axes as these are often noisy. Needs to be above 10% in both axes or the AP is 0
AP is averaged over 0.5, 1, 2, 4 meters (stricter to more lenient)

True positive metrics (after finding an object):
Are interpretable - native units - if ATE is 0.5 - you were off by 0.5m
10% recall-precision rule applies here as well - error set to 1.0

Average Scale Error (ASE) is the 3D intersection over union (IOU) after aligning orientation and translation. Only punishes size errors and not positioning - decoupling errors.

NDS:
50% - mAP
50% - sum of detection metric errors: ATE, ASE, AOE, AVE, AAE, 

sMOTA - scaled Multi Object Tracking Accuracy - scaled so it lies between 0 and 1
sAMOTA - 40 point threshold/recall point - range [0.1, 1] - interpolation average (increased from 11 in KITTI) - main metric for tracking

TID - Time between object appearing and getting detected
LGD - Longest duration of any detection gap in a track


BEVDet
4 parts: Image-view encoder, view transformer (makes BEV, NOT attention transformer), BEV encoder (extracts BEV features), head
Head - Uses only 1st stage of CenterPoint - heatmap

Custom data augmentation:
Image view is augmented with matrix A, but this interferes with its 3D
geometry as well in stage 2 (so further down, its shifted coordinates don't
match with the ground truth). To reverse this, A^-1 is used

BEV-space augmentation:
Now, further augmentation is done to the BEV space due to difference in data
between image and BEV in datasets like nuScenes. 
Spatial consistency: both 2D BEV and 3D ground truth are converted - we want the head to learn here which is why it's not reversed as before 
This is specific to BEVDet - using 2 kinds of augmentation - there's no attention mechanism between 2D and 3D or between the 4 different parts of BEVDet so this works

Scale-NMS:
Regular NMS sucks in BEV space. Scale-NMS inflates boxes depending on class.
For smaller boxes like for pedestrians, the box is temporarily inflated for
the NMS calculation for bigger overlap. 


BEVDet4D
Main additions: 
Extra encoder - processes concatenated maps from t-1 and t

Feature alignment - shifts previous BEV map so that ego at t-1 goes to where
the ego was at t (eg. move forward 5m, BEV is shifted back 5m)
Static objects: go back to where they were
Moving objects: appear as shifted in the direction they were going
The extra encoder sees this displacement, thus we get velocity

Bilinear interpolation problem: 
Most accurate method: For getting the shifted map for t-1, shift the ENTIRE point cloud of t-1 to match where the ego was when looking from t. Then calculate the BEV map from there again (lift-splat). This is extremely slow. You cannot use the previous lookup table because that is specific to the position it was in, not the shifted position.
Compromise: just shift the 2D BEV map from t-1 and combine it with t. Shifting here requires bilinear interpolation which can decrease precision.


BEVFormer
BEV queries, spatial cross-attention, temporal self-attention

SCA: 
Uses deformable attention, modified from 2D to 3D perception
BEV queries are lifted to a pillar, then reference points are sampled from this pillar, then projected to 2D views (through projection matrix of cameras). The cameras that capture the points in these 2D views are the hit views. Then, use deformable attention to grab features from around the pixels where the projections fell in the hit views

TSA:
BEVDet4D: Focuses on simplicity and speed. It aligns the past frame to the current ego-car position and stacks the features along the channel dimension. This allows the network to learn velocity by observing the "offset" between the same object across two concatenated frames.
BEVFormer: Focuses on rich feature interaction. Even after alignment, it acknowledges that objects move independently of ego-motion. Its Temporal Self-Attention layer helps the model "look" for corresponding object features in the history even if they have shifted due to their own velocity. Can more effectively model long temporal dependency.

Ironically, BEVFormer v2 went back to the BEVDet4D method of warp and concatentate.


Occupancy Networks
Takes p (3D coordinate) and x (photo)
Neural network f_theta(p, x) -> probability that spot p in x is full

Training:
The Sampling (pij): For every object (like a car), the model picks K random points in a box around it.

    Some points are inside the car (Occupancy = 1).

    Some are in the air around it (Occupancy = 0).

The Loss (LB): This is a standard Binary Cross-Entropy test.

If the model says "0.1 (Empty)" for a point that is actually inside the car (oij=1), the "Loss" is high, and the model adjusts its weights (θ).

The "Padding" Trick: They found that sampling just outside the box helps the model learn the "edges" of the object, so it doesn't get blurry.

Generative Modeling:
Imagine you have a blurry, partial photo of a car (like a car hidden behind a bush). A normal network might get confused. A Probabilistic Model (using a VAE - Variational Auto-Encoder structure) handles this by using a "Style Code" or Latent Vector (z).

The Encoder (gψ): This "Summarizer" looks at a bunch of points and their occupancies and says, "Based on these clues, this looks like a Sedan." It compresses that "Sedan-ness" into a small vector z.

The Mean (μ) and Sigma (σ): Instead of one fixed guess, it predicts a Distribution. It says, "I'm pretty sure it's a Sedan, but it could be a Hatchback."
KL-Divergence: This is a mathematical "Penalty" that keeps the model's guesses organized. It forces the "Sedan Code" to stay within a reasonable range so the model doesn't start hallucinating impossible 3D shapes.

Inference:
MISE, Marching Cubes, Fast-Quadric-Mesh-Simplification, refine using second order/gradient information - double backprop.


TPVFormer
Point querying: For (x, y, z) in the real world, project to the 3 planes, sample at projections, aggreagte the features from those samples. Sampling - bilinear interpolation, aggregation - summation

Cross-View Hybrid-Attention (CVHA) - Interactions among TPV queries - contextual information
Image Cross-Attention (ICA) - TPV queries - aggregate visual info from image features using deformable attention

Hybrid-Cross-Attention block (HCAB) - has CVHA and ICA - querying visual information effectively

Hybrid-Attention block (HAB) - only CVHA - specializes in contextual information encoding - faster than HCAB as it doesn't look at images

Image Cross-Attention: Same as SCA in BEVFormer - For a query (h, w), calculate the real world coordinates (x, y) using camera extrinsics/intrinsics, sample reference points coming "out" of that point, project these points onto the cameras, we get view hits again, use deformable attention on these - form updated TPV queries by summing up the sampled image features weighted by their attention weights

CVHA: Also uses deformable attention, for a point in any plane, it looks at the other 2 planes at that "same point" and uses deformable attention to extract information

TPVFormer is specifically a Hybrid Explicit-Implicit representation:
Explicit: The three 2D Planes are explicit (they are physical arrays).
Implicit: The relationship between those planes and the 3D points is implicit. You don't store 3D data; you store 2D data that implies 3D data through the summation and interpolation logic.

Limitation:
Loss of Axis-Specific Resolution: Each TPV plane collapses 3D information along a specific axis. This results in overlap ambiguity: Multiple object instances located along the same collapsed axis may be merged into a single feature. It doesn't have a mechanism to say that this merged view is garbage and should be disregarded.


SparseOcc
Sparse voxel decoder, mask transformer (unified segmentation - semantic, instance and panoptic concurrently), RayIoU
Fully sparse - doesn't depend on dense 3D feature, no sparse-to-dense and global attention operations

Sparse Voxel Decoder: 
Coarse-to-fine, only models non-free regions. Coarse voxel queries are equally distributed in 3D space, then upsampled to a 2x2 Rubik's cube, then pruned by threshold (above a certain occupancy score) or top-k selection (used in code).

Each layer receives voxel queries - 3D location in the real world (x, y, z) and content vector (C-dimensional) of features at that location. Self-attention is used to gather local and global features for the queries. Linear layer is used to generate 3D sampling offsets for each voxel query from the content vector- sample features from nearby coordinates to understand current voxel better. Then, 3D -> 2D for the reference points, intelligent weighing relevant features for the specific voxel (adaptive mixing).
Note: Offsets are generated from the content vector. If you only used the 3D location, the model would always look at the exact same spot in the images regardless of what is actually there. By using the content vector, the model can dynamically adjust its "eyes" to focus on the most informative parts of the image (like boundaries or textures) to refine the occupancy prediction.

Temporal modeling: Warp and concatenate doesn't work here due to inherent sparsity of 3D features in SparseOcc. Instead:
Current Point: "I'm looking at this 3D spot right now."
The Math: "Based on how I drove, this spot was there 0.5 seconds ago."
The Grab: "Let me look at the old image at that there location and grab the features."
The Mix: "Now I'll compare what I see now with what I saw then for that same physical spot." (uses adaptive mixing again)
The current reference point is changed to match what the reference point was in the previous frame.

Supervision: The model only grades the voxels it keeps (using BCE, as this is class-agnostic - only checks if they are occupied), and due to severe class imbalance, has higher loss for rarer objects

Mask transformer - for semantic labeling:
N queries - Qm - where, Qc - what.
3 steps: Multi-head self attention (for interaction between different queries), mask-guided sparse sampling, adaptive mixing.

Mask-guided sparse sampling - Initially, start with the coarse voxel mentioned before, then iteratively keep using the mask generated from the previous decoder layer. Pick random 3D points inside this mask, sample 2D images - extract features, bilinear interpolation, adaptive mixing. Keep refining from the previous mask. Temporal modeling using this is also easy.

Prediction - 
Uses queries for ->
Class prediction: passed through linear classifier, sigmoid
Mask prediction: passed through MLP -> mask embeddings - filters to search 3D space for the objects shape. Dot producted with sparse voxel embeddings - mask predictions. Important note: these then serve as mask queries for the next layer of mask-guided sparse sampling.

Here is the absolute full flow, from the first 2D image pixel to the final 3D occupancy voxel.
1. Stage 1: The "Scout" (Sparse Voxel Decoder)

    Input: 2D Image Features from your ResNet/Swin backbone.

    Process: It starts with a coarse grid (e.g., 25×25), upsamples by 2×, and uses Top-K Pruning to delete "Empty Air."

    Output: A set of K Sparse Voxel Embeddings (V). These are the "surviving" 3D points that might contain objects.

2. Stage 2: The "Artist" (Mask Transformer)

This stage has L layers (usually 3 or 6).
Layer 1 (The Initial Guess):

    Input: N random "Object Queries" (Content Vectors Qc).

    Initial Mask: Since there is no "previous" mask, it starts by looking at all K voxels from the Scout.

    Sampling: It projects those 3D points to the images, grabs features, and "mixes" them into the Qc vectors (forms mask embeddings M, them M . V).

    Prediction: It performs a Dot Product between the updated Qc and the K voxels.

    Result: It produces the first Mask Query (Qm)—a bitmask of which voxels belong to "Object #1," "Object #2," etc.

Layer 2 to L (The Refinement Loop):

    Input: The Qm (Masks) and Qc from the previous layer.

    Mask-Guided Sampling: Instead of looking at all K voxels, the model says: "Object #1 is inside this specific mask. Only sample 3D points from within that mask's volume."

    Bilinear Interpolation: It projects those refined 3D points onto the 2D images.

    Adaptive Mixing: It updates the Content Vector (Qc) with these "zoomed-in" high-res image features.

    New Prediction: It does the Dot Product again (M⋅V).

    Refined Mask: It outputs a sharper Qm for the next layer.

3. Stage 3: The "Final Verdict" (Output)

    Classification: The final Qc goes through a Linear Layer to say "This is a Bus."

    Occupancy: The final Qm tells you exactly which of your K sparse voxels form that bus.

Supervision: Missed detection - if objects are absent due to predicted sparse occupancy, ground truth for that is discarded. For false positives, categorize as "no object" category (as "background" label in Hungarian matching)

Loss functions: Focal loss, DICE loss - if blobs overlap well, Mask loss - if fine detail voxels are right, Occ loss - for sparse voxel decoder - if the predicted sparse geometry is good.
Loss only calculated for pairs matched with Hungarian Matching

RayIoU:
Problems with mIoU:
Visible mask needed as some areas scanned by lidar are marked as free leading to label inconsistency. Visible mask is also bad as mIoU can be hacked by predicting thicker surfaces to get higher mIoU scores. Dense methods like BEVFormer can achieve this by using a visible mask - the area behind the surface lacks supervision. 
Penalty is strict for thin surfaces resulting in IoU of zero.
No temporal factor, only current moment is considered, reducing occupancy prediction to a depth estimation task.

RayIoU: Emulates LiDAR behavior by projecting query rays into the predicted 3D occupancy volume. "For each query ray, we compute the distance it travels before intersecting any surface and retrieve the corresponding class label. We then apply the same procedure to the ground-truth occupancy to obtain the ground-truth depth and class label. In case a ray does not intersect with any voxel present in the ground truth, it will be excluded from the evaluation process."
More forgiving: Query ray is a true positive if class labels coincide and the L1 error is less than a threshold value.
Temporal casting: Better evaluates scene completion performance by making it so that the query ray can originate from the current, past or future. The Test: If your model "completed" the back of the bus correctly in the current frame, the future ray will hit it exactly where the future LiDAR says it should be.
